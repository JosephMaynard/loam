# 13 ‚Äî Internationalization (multi-lingual UI)

> **Status: implemented (admin-selected, node-wide).** The UI chrome, server error strings, and the
> full-screen gate states are now translated into **15 languages** ‚Äî English (source) plus Spanish,
> French, Arabic, Farsi/Persian, Portuguese, Ukrainian, Russian, Turkish, Burmese, Urdu, Dari, Pashto,
> Swahili, and Bengali (five RTL: `ar`, `fa`, `ur`, `prs`, `ps`). The rest of this doc still describes
> the original design; the notes below record how the shipped version differs.
>
> **Shipped design (differs from the draft below):**
> - **Locale is admin-selected for the whole node**, not per-user. The admin picks one language in the
>   Admin ‚Üí Identity panel; it is stored as `node.locale` (`packages/schema` `LocaleSchema`), surfaced
>   in `NetworkConfig`, and applied to every user. Editing it broadcasts `configUpdated` and every
>   connected client re-renders in the new language (flipping `<html dir="rtl">` for RTL locales)
>   without a reload. There is **no per-user picker** ‚Äî the resolution logic in ¬ß3 below is superseded.
> - **The catalog lives in `apps/client/src/i18n/`**: `en.ts` (source of truth + `Catalog` type), a
>   tiny `t()`/`errorText()` runtime in `index.ts`, and one file per locale. A completeness test
>   (`i18n.test.ts`) fails the build if any locale is missing a key, has the wrong CLDR plural
>   categories, or introduces a stray `{token}`. Server errors carry a stable snake_case `code`
>   (`ERROR_CODES` in `apps/server/src/app.ts`) alongside the English `error`; the client localizes by
>   code and falls back to the English string for unknown codes (forward-compatible with a mixed-version
>   mesh).
>
> **‚öÝÔ∏è Translation quality caveat.** The 14 non-English catalogs are **machine-translated (by an LLM)
> and NOT yet field-verified** ‚Äî every locale file carries a header saying so. They are a strong
> starting point, not a substitute for native review: RTL phrasing (Arabic/Farsi/Urdu/Dari/Pashto),
> Slavic plural categories, and the lower-resource languages (Burmese, Pashto, Dari, Swahili) most want
> a native pass before high-stakes field use. Because the locale is **admin-selected**, an operator only
> ever exposes a language they trust, and `en` is the compile-time-guaranteed fallback, so no key can
> render blank.

LOAM's README promises multilingual support. Today that promise is half true: people can *type* in any
language and it renders correctly, but every button, label, error, and toast the app itself produces is
hardcoded English. This is the plan to close that gap without betraying the off-grid, low-bandwidth
constraints that make LOAM LOAM.

## Translation policy

The 15 shipped languages (English plus Spanish, French, Arabic, Farsi/Persian, Portuguese, Ukrainian,
Russian, Turkish, Burmese, Urdu, Dari, Pashto, Swahili, and Bengali) were chosen for humanitarian
coverage and accessibility, not popularity or market size. Several are languages spoken by populations
disproportionately likely to need offline, local communication tools ‚Äî people displaced by conflict or
disaster, refugee and camp settings, and multilingual emergency response ‚Äî alongside languages with
large numbers of speakers and RTL scripts that exercise LOAM's layout support.

As noted below, the 14 non-English catalogs are machine-translated and have not yet had a
native-speaker review; treat them as a strong starting point, not a finished translation, and give
extra scrutiny before relying on one for a high-stakes or public-facing deployment. Native-review
contributions are welcome ‚Äî see the completeness checks in `i18n.test.ts` for what a corrected or new
catalog needs to satisfy.

Shipping a language is a statement about accessibility, not about any government, conflict, or party to
one. The inclusion or omission of a given language carries no endorsement, position, or comment on any
group, state, or conflict ‚Äî it reflects only an assessment of who is likely to need an offline
communication tool, and in what language.

## 1. What already works ‚Äî and what doesn't

**Works today:**

- **Message content in any language, including RTL.** `dir="auto"` on message bodies, composers, and
  search inputs (four sites) lets the browser pick direction per-string, so Arabic/Hebrew/Farsi text
  renders correctly interleaved with Latin. This is the [07](07-more-features.md) RTL work; it is real
  and it is enough for *content*.
- **Dates and times via `Intl`.** `dayLabel()` (`src/lib/dates.ts`) and the message timestamp
  formatter (`app.tsx`) both use `new Intl.DateTimeFormat(undefined, ‚Ä¶)`, so the *format* already
  follows the browser's locale. The engine ships every locale's date rules; no data to bundle.

**Does not translate (the gap this doc closes):**

| Surface | Where | Example |
|---|---|---|
| **All UI chrome** | `apps/client/src/app.tsx` | "Settings", "New message in", "Waiting for approval", empty-state copy, composer placeholders, admin/moderation labels |
| **Server error strings** | `apps/server/src/app.ts` | "Direct messages are disabled on this LOAM node", "Admin access required" ‚Äî **76 distinct** English strings returned in `{ error }` |
| **The two hardcoded date words** | `src/lib/dates.ts` | `"Today"` / `"Yesterday"` ‚Äî English literals inside an otherwise-`Intl` function |
| **Toast + OS-notification copy** | `app.tsx` (`pushToast`, `notifyIfHidden`) | "New message in #general" ‚Äî the label is chrome; the body (message content) is user text and stays as-is |
| **Generated display names** | `packages/display-name` | `adjective.material.creature` from English word lists ‚Äî see ¬ß4, deliberately left English |

## 2. Approach ‚Äî a hand-rolled typed catalog + error codes

LOAM's constraints rule out most of the off-the-shelf playbook:

- **Offline-first.** No CDN, no runtime locale fetch. Every locale a node might serve must be *in the
  bundle* the node already serves over the LAN. There is no "lazy-load `fr.json` from the network"
  fallback that a normal web app leans on.
- **Tiny bundle budget.** The client is served over a phone hotspot to people on the same slow radio.
  A heavy i18n runtime is a tax every joiner pays on first load.
- **No build-step gymnastics.** The house style is plain `tsc`/Vite with no codegen; a catalog
  compiler or macro transform is friction we should avoid unless it earns its place.
- **Preact.** Rules out React-only ecosystems and anything assuming the React reconciler.

### Options weighed

**(a) Hand-rolled typed catalog ‚Äî recommended.** A `t(key, params?)` helper over plain typed objects,
one module per locale (`src/i18n/en.ts`, `src/i18n/ar.ts`, ‚Ä¶). `en` is the source of truth and the
type; every other locale is typed as `Record<keyof typeof en, string>`, so a missing key is a
*compile error*, not a silent English leak. Interpolation is `{name}`-style token replacement in ~15
lines. Cost: ~1‚Äì2KB of helper code plus the catalog strings themselves (which any approach pays).
Zero runtime dependency, no build step, tree-shakeable per locale if we ever split. This is the
natural fit and matches the "named functions, validate at boundaries, no store lib" grain of the
codebase.

**(b) An i18n library.** `preact-i18n` is small but unmaintained and untyped; `i18next` is ~40KB+
min and brings a plugin/backend model built for the CDN world we don't live in; `lingui` gives
excellent ICU + extraction but needs a **Babel/SWC macro build step** and an ICU runtime ‚Äî exactly
the build gymnastics and bundle weight we're avoiding. For LOAM's string count (low hundreds), a
library's machinery costs more than it saves.

**(c) Server errors: codes, not translated strings ‚Äî recommended alongside (a).** The server should
**not** try to localize its error prose ‚Äî it doesn't know the reader's locale, and in the mesh case
([11](11-node-sync.md)) two nodes may run *different versions* with different English wording. Instead
add a stable machine-readable `code` to the error envelope and keep the English `error` as a
human/debug fallback:

```jsonc
// today:            { "error": "Direct messages are disabled on this LOAM node" }
// proposed:         { "error": "Direct messages are disabled on this LOAM node",
//                     "code": "dms_disabled" }
```

The client maps `code` ‚Üí a catalog key and renders in the reader's locale, falling back to the
server's `error` text when it doesn't recognise the code (older client vs newer server, or vice
versa). The 76 strings become ~76 codes ‚Äî a mechanical pass. This also makes errors *testable* as
identifiers rather than brittle string matches, and it's forward-compatible with sync across mixed
versions.

**Recommendation: (a) + (c).** A typed catalog for the client, error codes from the server localized
client-side. Reject the libraries; the constraints that make LOAM interesting are the ones they're
built to ignore.

## 3. Locale selection

> **Superseded by the shipped design.** The per-user resolution below (localStorage override ‚Üí
> `navigator.languages` ‚Üí `en`) was **not** built. The node runs a single admin-selected locale
> (`node.locale`, resolved server-side and delivered via `/api/config`); there is no `loam-locale`
> localStorage key and no `navigator.languages` walk. The "follow-up (not v1)" note at the end of this
> section ‚Äî a per-node default ‚Äî is in fact what shipped. Kept for design history only.

Resolution order, first match wins:

1. **Explicit user override** ‚Äî a locale picker in Settings, persisted to `localStorage`
   (`loam-locale`). This is the authoritative choice once set.
2. **`navigator.languages`** ‚Äî walk the browser's ordered preference list, take the first that has a
   catalog; fall back to the base language (`ar-EG` ‚Üí `ar`).
3. **`en`** ‚Äî the guaranteed-present source-of-truth catalog.

**Follow-up (not v1):** a per-node default. `node` config today is just `{ name }`
(`NodeIdentityConfigSchema`); adding `node.locale` lets an operator ship a node that greets joiners in
the local language before they touch a setting. Surface it through `/api/config` like the other node
identity fields. Small, additive, deferrable.

## 4. The display-name question ‚Äî decided: stay English (untranslated)

`packages/display-name` turns an id into `adjective.material.creature` (e.g. `swift.copper.otter`)
via FNV-1a hashing over **English word lists**, and it must be **deterministic across every client** ‚Äî
that's the whole point: two people looking at `user.1a2b3c4d` must see the *same* name so the name
functions as a shared identifier. Translating the word lists would break that: a French client and an
English client would hash to different-language names for the same id and could no longer refer to "the
copper otter" to each other.

**Decision: display names are proper-noun-like identifiers and stay in the English word-list form,
untranslated.** They are treated like a username, not like UI copy. Documented here so nobody "fixes"
it later. (A future scheme could hash to language-neutral tokens, but that's a data-model change, not
i18n, and out of scope.)

## 5. Pluralization & interpolation ‚Äî `Intl`, no ICU

Interpolation is positional token replacement (`t("unread_count", { n })` ‚Üí `"{n} unread"`). For
plurals, use **`Intl.PluralRules`** ‚Äî the engine already knows every language's plural categories
(`one`/`few`/`many`/`other`). The catalog stores a small object per pluralized key:

```ts
unread: { one: "{n} unread message", other: "{n} unread messages" }
// t() picks the branch via new Intl.PluralRules(locale).select(n)
```

No ICU MessageFormat library, no `{count, plural, ‚Ä¶}` mini-grammar to parse. The pluralized keys are a
handful (unread counts, member counts, "N replies", "N people waiting"); `Intl.PluralRules` covers
them with zero bundle cost.

## 6. Scope, extraction, and testing

**String count.** `app.tsx` (~5,200 lines) is the whole UI in one module. A grep for JSX text nodes
finds ~120 literal text strings; add ~25 `placeholder`/`aria-label`/`title` prop strings, plus toast
and notification labels ‚Äî call it **~250‚Äì300 distinct UI strings** to extract, on the order of a
hundred keys once near-duplicates collapse. Server side: **76** error strings ‚Üí 76 codes. This is a
bounded, countable job, not an open-ended one.

**Extraction strategy ‚Äî mechanical, per-view.** Work one view at a time (Settings ‚Üí Admin ‚Üí
People/moderation ‚Üí composer/empty states ‚Üí toasts), lifting literals into `en.ts` keys named by
locality (`settings.title`, `moderation.banConfirm`). Doing it per-view keeps each diff reviewable and
lets a locale-completeness test gate progress. `dates.ts`'s `"Today"`/`"Yesterday"` fold into the
catalog in the same pass.

**Test strategy.** Add an i18n test (Vitest, alongside the existing `src/lib/*.test.ts`) that asserts
**every non-`en` catalog has exactly the `en` key set** ‚Äî no missing keys, no strays. Because non-`en`
catalogs are typed as `Record<keyof typeof en, string>`, `tsc` already catches missing keys at build
time; the runtime test also catches *extra*/stale keys and the plural-object shape. Add one test that
every server error `code` has a catalog entry, so a new server error can't ship without a translation
key.

**RTL layout audit (still needed).** `dir="auto"` fixes *content* direction but not *layout* ‚Äî the app
shell, sidebars, and chrome are laid out with **physical** CSS properties that don't flip for RTL. All
of these live in `src/global.css` and should move to **logical properties**
(`margin-inline-start`, `border-inline-end`, `inset-inline-end`, `text-align: start`):

| `global.css` line | Selector | Physical property ‚Üí logical |
|---|---|---|
| 120 | `.sidebar` | `border-right` ‚Üí `border-inline-end` |
| 244 | (text block) | `text-align: left` ‚Üí `start` |
| 489 | `.markdown-body ul` | `padding-left` ‚Üí `padding-inline-start` |
| 493 | `.markdown-body blockquote` | `border-left` ‚Üí `border-inline-start` |
| 495 | `.markdown-body blockquote` | `padding-left` ‚Üí `padding-inline-start` |
| 568 | `.message-delete` | `margin-left: auto` ‚Üí `margin-inline-start: auto` |
| 697 | `.thread-panel` | `border-left` ‚Üí `border-inline-start` |
| 707 | `.thread-header > div` | `margin-right: auto` ‚Üí `margin-inline-end: auto` |
| 752 | `.connection-error` | `right: 16px` ‚Üí `inset-inline-end` |
| 1111 | `.toast-stack` | `right: 16px` ‚Üí `inset-inline-end` |
| 1127 | (text block) | `text-align: left` ‚Üí `start` |
| 1361 | `.conversation-header-actions` | `margin-left: auto` ‚Üí `margin-inline-start: auto` |

Twelve sites ‚Äî small and mechanical. `text-align: start` is already used once in the file, so the
pattern is established. Set `<html dir="rtl" lang="‚Ä¶">` from the resolved locale (a short RTL-language
allowlist), and the logical properties do the rest.

## 7. Effort

**M.** No new dependency, no build step. The work is: the ~15-line `t()` helper + `Intl.PluralRules`
plumbing; the per-view extraction into `en.ts` (~a hundred keys); the server `code` pass (76); the
locale picker + `localStorage`/`navigator.languages` resolution; the 12-site logical-property CSS
sweep and `<html dir>` wiring; and the completeness tests. Adding each *additional* language after the
scaffolding is just a new typed catalog module ‚Äî the framework cost is paid once.

## Decisions recorded here

- **Client i18n = hand-rolled typed catalog**, not a library. `t(key, params)` over plain per-locale
  modules; `en` is the type and the fallback; missing keys are compile errors. Rejected `i18next`
  (bundle + CDN model), `lingui` (macro build step + ICU runtime), `preact-i18n` (unmaintained,
  untyped).
- **Server errors carry a stable `code`; localization happens client-side.** Keep the English `error`
  string as a fallback. Chosen for the mixed-version mesh case and for testability.
- **Pluralization uses `Intl.PluralRules`**, interpolation uses positional `{token}` replacement ‚Äî no
  ICU MessageFormat library.
- **Locale resolution:** Settings override (`localStorage`) ‚Üí `navigator.languages` ‚Üí `en`. Per-node
  `node.locale` default is a deferred follow-up.
- **Generated display names stay in the English word-list form, untranslated** ‚Äî they are shared
  deterministic identifiers; translating them would break cross-client identity.
- **RTL layout** is finished by migrating the 12 physical-property sites in `global.css` to logical
  properties and setting `<html dir>` from the locale; `dir="auto"` already handles content.
